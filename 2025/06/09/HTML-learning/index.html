indexedDb

概述

现有浏览器数据储存方案不适合存储大量数据，cookis的大小不超过4KB且每次请求都会发送回服务器，LocalStorage在2.5MB到10MB之间，且不提供搜索功能，不能建立自定义的索引。
indexedDB就是浏览器提供的本地数据库，可以被网页脚本创建和操作。indexedDB允许储存大量数据。提供查找接口，还能建立索引。不属于关系型数据库，不支持sql查询语句，更接近NoSQL数据库。

indexedDB的特点：
1.键值对储存。indexedDB内部采用对象仓库存放数据，所有类型的数据都可以直接存入，包括JavaScript对象。对象仓库中，数据以”键值对“的形式保存，每一个数据记录都有对应的主键，主键是独一无二的，不能有重复，否则会抛出一个错误。
2.异步。indexedDB操作时不会锁死浏览器，用户依然可以进行其他操作。这与localStorage形成对比，后者的操作是同步的。异步设计是为了放置大量数据的读写，会拖慢网页的性能。
3.支持事务，indexed DB支持事务，这意味着一系列操作步骤之中，只要有一步失败，整个事务就都取消，数据库回滚到事务发生之前的状态，不存在只改写一部分数据的情况。基于事务的一致性和原子性，保障数据的准确和可靠性。
4.同源限制IndexedDB受到同源限制，每一个数据库对应创建它的域名。网页只能访问自身域名下的数据库，而不能访问跨域的数据库。
5.储存空间大，indexedDB的储存空间比localStorage大得多，一般来说不少于250MB，甚至没有上限。
6.支持二进制储存。indexedDB不仅可以储存字符串，还可以储存二进制数据(Array Buffer对象和Blob对象)。

基本概念

indexedDB是一个比较复杂的API，涉及不少概念。它把不同的实体，抽象成一个个对象接口。学习这个API，就是学习它的各种对象接口。

  • 数据库:IDBDatabase对象
  • 对象仓库：IDBObjectStore对象
  • 索引:IDBIndex对象
  • 事务：IDBTransaction对象
  • 操作请求：IDBRequest对象
  • 指针：IDBCursor对象
  • 主键集合：IDBKeyRange对象

主要概念：

1.数据库：
数据库是一系列相关数据的容器。每个域名都可以新建任意多个数据库。
IndexedDB数据库有版本的概念，同一个时刻，只能有一个版本的数据库存在。如果要修改数据库接口(新增或删除表、索引或者主键)，只能通过升级数据库版本完成。
2.对象仓库
每个数据库包含若干个对象仓库。类似于关系型数据库的表格。
3.数据记录
对象仓库保存的是数据记录。每条记录类似于关系型数据库的行，但是只有主键和数据体两部分。
主键用来建立默认的索引，必须是不同的，否则会报错。主键可以是数据记录里面的一个属性，也可以指定为一个递增的整数编号。

4.索引
未来加速数据的检索，可以在对象仓库里面，为不同的属性建立索引。
5.事务
数据记录的读写和删改，都要通过事务完成。事务对象提供error，abort和complete三个时间，用来监听操作结果。

webworker

学习后感受

Web Worker 本质上是浏览器暴露给 JavaScript 的线程管理接口，通过沙箱机制确保安全。
1.webworker是浏览器主线程分出来的子线程，无法直接对window对象进行直接操作，同理也无法访问document，parent等主线程对象。worker中的全局变量是self,但self并不完全等同于window
2.webworker对于某些第三方库的引入需要使用特殊的方法（例如使用importScriptes，或者使用模块Worker），有时候甚至需要修改第三方库的底层代码。PS:也是因为没有办法直接操作window导致的，因为大部分的第三方库是基于浏览器主线程的，底层代码中有时候会直接写window，而webworker子线程没有办法直接使用，会导致报错。
3.使用webworker，主文件使用postMessage进行操作必须传递一个参数给webworker文件，但是webworker不一定将处理的结果传递给主文件，若需要则使用onMessage进行获取。
4.可以创建一个共享的webworker，连接共享worker通信必须通过port对象，给共享worker提供一个确切的打开端口供脚本与worker通信。
5.主线程与子线程之间的变量通信并不是通过共享实现的，通过拷贝相同的值来实现，不是共享同一个实例。拷贝而并非共享的那个值称为消息(message)可以使用 postMessage() 将消息传递给主线程或从主线程传送回来。message 事件的 data 属性就包含了从 worker 传回来的数据。
6.WebWorker的同源限制：worker脚本必须与主线程的页面同源。
7.子线程中无法直接操作本地文件系统，需通过file API间接处理。
8.浏览器能直接对线程进行调试，当创建线程对象后会自动创建线程用于调试。

webSocket

基本概念

websocket是一种协议，用于Web应用程序和服务器之间建立实时，双向的通信连接。通过一个单一的TCP连接提供了持久化连接，使得Web应用程序可以更加实时地传递数据。WebSocket协议最初由W3C开发，并于2011年成为标准。
Websocket协议是一种基于TCP的协议，用于在客户端和服务器之间建立持久连接，并且可以在这个连接上实时地交换数据。有自己的握手协议，用于建立连接，也有自己的数据传输格式。

  • 优势
    实时性：由于WebSocket的持久化连接，可以实现实时的数据传递，避免了Web应用程序需要不断地发送请求以获取最新数据的情况。
    双向通信：WebSocket协议支持双向通信，意味着服务器可以主动向客户端发送数据，而不需要客户端发送请求。
    减少网络负载：由于WebSocket的持久化连接，可以减少HTTP请求的数量，从而减少了网络负载。

劣势：
需要浏览器和服务器都支持：Websocket是一种相对新的技术，需要浏览器和服务器都支持。一些旧的浏览器和服务器可能不支持WebSocket.
需要额外的花销：WebSocket需要在服务器上维护长时间的连接，这需要额外的花销，包括内存和cpu
安全问题：由于WebSocket允许服务器主动向客户端发送数据，可能会存在安全问题。服务器必须保证只向合法的客户端发送数据。

SSE Server-Sent Events

基本概念

常规的http请求，发送的数据为数据包，当数据包发送完成后，则会断开http请求。所以想要通过HTTP协议做到服务器主动推送信息，则需要服务器向客户端声明，接下来要发送的是流信息，而非数据包，会连续不断的发送数据，所以客户端不会关闭连接，会一直等着服务器发过来的新的数据流。视频播放就是这样的例子。SSE就是利用这种机制，使用流信息向浏览器推送信息。基于HTTP协议，目前除了IE/Edge，其他浏览器都支持。

优点：
SSE 使用 HTTP 协议，现有的服务器软件都支持。WebSocket 是一个独立协议。
SSE 属于轻量级，使用简单；WebSocket 协议相对复杂。
SSE 默认支持断线重连，WebSocket 需要自己实现。
SSE 一般只用来传送文本，二进制数据需要编码后传送，WebSocket 默认支持传送二进制数据。
SSE 支持自定义发送的消息类型。

三种连接方式汇总

自述：HTTP：客户端发送请求，服务端发送数据包，断开连接。
补充：HTTP 1.1/2/3：默认启用持久连接（Connection: keep-alive），请求响应后不会立即断开，可复用连接。
断开时机：连接可能因超时、显式关闭（Connection: close）或服务器策略而断开。
无状态：每个请求独立，服务器不保留客户端状态（需通过 Cookie/Session 等机制实现状态管理）

自述：websocket：客户端发送连接请求，服务端接收后发送响应请求，客户端接收到再次发送连接请求，服务端发送数据。不会断开。
补充：握手流程：
客户端发送 HTTP 请求（携带 Upgrade: websocket 头）。
服务器返回 101 状态码（协议升级），双方建立 TCP 连接。
之后不再需要 HTTP 协议，直接通过 WebSocket 帧通信。
持久连接：默认不断开，但可通过 close() 方法主动关闭。
双向通信：客户端和服务器均可随时发送数据，无需等待请求。

自述：SSE：客户端发送请求，服务端发送数据，本质是一次耗时很长的HTTP下载请求。
补充：单向通信：仅服务器向客户端发送数据，客户端无法主动推送（需新请求）。
协议特性：
基于 HTTP 长连接，使用 text/event-stream MIME 类型。
服务器通过 data: …\n\n 格式持续发送数据，客户端通过 EventSource API 接收。
自动重连：内置重连机制（retry 字段），HTTP 连接中断后会自动恢复。
断开条件：请求完成（服务器关闭连接）或客户端主动终止。

什么是mvc模式

自我总结
1.用户向服务器发送请求与获取数据（前端点击触发获取数据的事件，通过axios发送请求）
2.服务器将该请求发送给处理数据的控制器（工具为jQuery，axios等）
3.控制器要求处理数据的模型返回数据(即后端常见的model，例如user model)
4.模型将在数据库中查找所有数据返回给控制器（数据库连接与sql查询）
5.1若成功，控制器要求与数据相关的试图返回数据的展示（返回200状态码同时返回数据）
5.2若模型返回错误，将通过要求处理错误的视图来展示(即404，403页面错误)
6.该视图从控制器中获取数据，并将数据展示在浏览器，将展示返回给控制器（前端获取数据后，生成对应的前端页面）
7.控制器将数据的表示返回给用户从而结束请求（将前端页面发送给浏览器，浏览器渲染）

模型是处理数据验证，逻辑和持久性
控制器是在模型和视图之间传递数据
视图则是用于处理怎样显示信息

优化后的版本：（vue举例）
1.用户交互，在视图中触发点击事件
2.控制器响应：methods中的方法被调用
3.模型草错，调用vuex action
4.视图更新 响应式数据自动驱动视图更新
5.通过axios与服务器通信

vue的双向绑定原理

vue的双向绑定是采用数据劫持结合发布订阅模式的方式，通过Object.defineProperty()来劫持各个属性的setter，getter，在数据变动时发布消息给订阅者，触发相应的监听回调。
具体步骤：
第一在create生命周期钩子之前，会将需要observe的数据对象进行深度遍历，将其每个对象中的属性都加上setter和getter，当给对象的某个属性赋值时，触发setter函数，就能监听到数据变化。
第二将模板中的变量替换成数据，然后初始化渲染页面视图，并将每个指令对应的节点绑定更新函数，添加监听数据的订阅者，一旦数据有变动，收到通知，更新视图。

  • 什么是原型
    每个函数都有自己的prototype属性,称之为原型,又因为这个属性为Object类型,所以也称之为原型对象.
    函数中定义的一些方法都保存在函数的prototype原型中,共享给实例对象使用.
    原型可以继承.

  • 什么是原型链
    对象都有__proto__属性,这个属性指向它的原型对象.原型对象也是对象,也有__proto__属性,指向原型对象的原型对象,这样一层一层形成的链式结构称为原型链,最顶层找不到则返回null

为什么arr能拿到Array中的方法呢?因为这些方法存放在Array的原型对象上,原型是支持继承的.
arr 是 Array 构造函数的实例，因此 arr.proto 自动指向 Array.prototype（这是 JavaScript 的原型继承机制）.当我调用arr.reverse()时,引擎在arr对象自身上查找reverse方法(未找到),接着通过arr.__proro__向上查找Array.prototype,在Array.prototype中找到了reverse方法,于是调用它.
Array.prototype 存储了所有数组共享的方法（如 reverse, push 等）。所有数组实例通过原型链继承这些方法，无需在每个实例上重复创建，节省内存并保持一致性

    异步任务则是当执行异步任务时，跳过异步任务执行异步任务后的任务，主线程不进行异步任务的执行，而是将其存放在消息队列中，当任务队列通知主线程，某个异步任务可以执行了，该任务才会进入主线程执行。

  • post请求和get请求有什么区别
    1.传参放置的位置不同，get使用url传参，post传参可以放在body请求体中传递。也因此不安全
    2.当浏览器发起跨域 POST 请求且请求满足以下任一条件时，会自动先发送一个 OPTIONS 请求（预检请求）：使用了自定义HTTP头（如 Authorization, X-API-KEY）Content-Type 不是以下三种之一：application/x-www-form-urlencoded multipart/form-data text/plain使用了非常规 HTTP 方法（如 PUT, DELETE 等）
    3.get请求可以被缓存，post默认不缓存
    4.get请求一般用于数据获取，数据包的大小有限，而post请求数据包无理论限制

  • CSRF 和 XSS 的区别
    区别一：
    CSRF ：需要用户先登录网站 A ，获取 cookie
    XSS ：不需要登录。
    区别二：（原理的区别）
    CSRF ：是利用网站 A 本身的漏洞，去请求网站 A 的 api 。
    XSS ：是向网站 A 注入 JS 代码，然后执行 JS 里的代码，篡改网站 A 的内容。
